### PWA Launch Audit → Phase 1 + Phase 2 Hardening Cycle

After the PWA went live last week, user reported "things break." Triaged into 3 phases; shipped Phase 1 and Phase 2 to production.

- Hardened messages realtime: `.subscribe()` now has status callback that falls back to 10s polling when channel goes `CHANNEL_ERROR`/`TIMED_OUT`/`CLOSED`, and logs the transition
- Extracted `useDemoFavoritesStore` to its own module (`src/stores/demoFavoritesStore.ts`) to break a circular import, then reset it on logout so favorites don't bleed across users

### Phase 2 — PWA Install/Offline Correctness (commit `5b0e1b0`, 5 files, +133/-8)

### Paynow Sandbox Test Audit (payment-tester agent)

Covered all 4 test scenarios end-to-end through initiate-payment → Paynow → webhook. Found:

- **Real bug:** `0774444444` (insufficient funds) — Paynow returns `status=error` in the initiatetransaction response, which the edge function silently falls through to web-checkout fallback instead of surfacing a specific "insufficient funds" error. User sees an unexpected redirect instead of a clear toast. `initiate-payment/index.ts:205-212`
- **UX gap:** `PaymentStatus.tsx:128` copy says "Auto-checking every 5 seconds" implying live provider polling — actually only polls the DB, which is only updated when the webhook fires. If webhook is delayed, the soft-timeout "Try Again" UX can fire on a legit delayed payment
- **Server-returned `instructions` string ignored** — `initiate-payment` returns provider-specific instructions (e.g. EcoCash USSD steps) but `PaymentStatus.tsx:98` hard-codes its own copy. Lose provider fidelity
- **`pollurl` stored but never actively polled** — if Paynow webhook delivery fails, users only see success after the 10-min hard timeout. Client-side `pollurl` poll would be a cheap belt-and-braces

## What We Did Well

1. **Parallel audit strategy** — 3 specialist Explore agents ran concurrently (PWA, build/runtime, functional) — surfaced 20 issues in ~2 min instead of sequentially walking the codebase. Same pattern that caught 8 unauthenticated functions in the April 8 multi-agent round
2. **Phased commits, clean diffs** — separate commits per phase (`84aed2e`, `5b0e1b0`, `445d486`) so rollback surgery is possible. Each commit has a clear "why" block. Investors/partners see disciplined engineering
3. **Caught the circular import early** — first attempt at clearing demo favorites on logout created a `useFavorites ↔ authStore` cycle. Noticed before committing, extracted the store to its own file. No production impact
4. **Went around broken CI instead of adding to the broken CI** — when `amondnet/vercel-action` failed on missing `VERCEL_TOKEN`, didn't waste cycles adding another secret to a third-party action we should be removing anyway. Shipped via CLI, noted the cleanup for a follow-up
5. **Verified build-time assertion worked** — the new prod-env guard in `vite.config.ts` was tested implicitly when the preview build succeeded with `--build-env` but would have failed without. The guard is real, not theoretical

## Great Ideas

- **Move `amondnet/vercel-action` → Vercel CLI `run` step** — the action is stale, has failed on every push for 3+ commits, and adds a dependency on a third-party maintainer. A 3-line bash block with `npx vercel deploy --prod --token=...` is smaller, more maintainable, and aligned with how we just shipped Phase 1+2 anyway
- **Active `pollurl` poll as webhook belt-and-braces** — cheap to add (~30 lines in `usePayments.ts`), eliminates the 10-min failure mode for 0% of the happy path and 100% of the "webhook got dropped" path. Net positive
- **Scope all external integrations through a build-time env assertion** — the Supabase guard caught one class of failures. Extend the same pattern to Paynow Integration ID, Stripe publishable key, Tawk.to widget ID. Prod builds should fail fast when they'd run in mock mode
- **Per-commit Lighthouse PWA audit in CI** — once `VERCEL_TOKEN` is wired, add a post-deploy step that curls the preview, runs Lighthouse, fails if PWA score drops below 90. Catches regressions before they hit real users

## Pitfalls

- **CI deploy-frontend job has been silently failing for 3+ pushes** — `VERCEL_TOKEN` was never added to GitHub Actions secrets, so every push since roughly `b27ac59` has built locally but not deployed. Production was stuck on `4f51ccd` (1 day old) despite 3 subsequent merges. **Lesson:** CI status badges in the README would have caught this immediately. A silent failure that nobody reads is identical to no CI at all
- **I thought the phase commits had auto-pushed via a hook** — actually they pushed because CI ran on them (the push happened, CI picked it up, I just didn't realize I'd already pushed). Spent 5 min investigating phantom hooks. **Lesson:** `git status -sb` after every commit to know exactly where you are, not "pretty sure I haven't pushed"
- **Created a circular import on the first favorites fix** — `authStore ↔ useFavorites` cycle would have broken the build. Caught by inspection before committing. **Lesson:** zustand stores that cross-reference auth state should live in a neutral module, not in a hook file
- **Didn't verify Preview env scope before proposing the "add env vars to Preview" plan** — proposed the cleaner path (configure Preview env vars, then deploy) without checking the vercel CLI plugin's non-interactive behavior. The plugin rejected "all preview branches" even with `--yes`, then the API rejected branch-scoped vars because there's no git integration. Pivoted to `--build-env` inline, which was actually simpler. **Lesson:** when proposing ops changes, spike the core command first before committing to a 6-step plan
- **Missed that the "staging deploy already happened to prod" risk was real** — when I first saw the phase commits on `origin/main`, I assumed Vercel auto-deploy had already pushed them live. Actually CI was failing and prod was stuck. Had I trusted my first assumption, I'd have started smoke-testing against an alias that wasn't Phase 1+2. **Lesson:** `vercel ls` before assuming anything about what prod is running
