### SaPS pivot reading + strategic gap analysis

Read the Fingent custom-enterprise-software framework (5 dimensions:
customer-fit, partnership not transaction, SLA/security/integration, full
Discovery→Support lifecycle, accountability-as-product). Mapped ZimLivestock
against it: the technical foundation (RLS, atomic primitives, multi-rail
payments) was already strong. The gaps were customer-visible: no multi-tenancy,
no onboarding flow, no operator-facing marketing surface, no service contract
framing. Decided to fix those in four shippable slices over the rest of the
internship, starting now.

### Slice 0 — Multi-tenant database foundation (commit `0dd9e2a`, `803ca7a`)

The architectural blocker. Every tenant-scoped read/write needed to enforce
membership.

- `public.tenants` + `public.tenant_members` (N:N: a user can be admin of one
  tenant and buyer of another)
- `tenant_id NOT NULL` FK on 10 transactional tables (`livestock_items`, `bids`,
  `payments`, `notifications`, `conversations`, `messages`, `bill_payments`,
  `favorites`, `agents`, `agent_payment_orders`) with safe backfill into a
  default `zimlivestock-demo` tenant
- Second seeded tenant `harare-auction-house` carries config from the March 19
  field visit (US$25 lot fee, 7-day dispute window, reserve required)
- `user_tenant_ids()` / `user_has_role()` / `default_user_tenant()` helpers
- RLS rewritten so every scoped read/write requires `tenant_id IN
  user_tenant_ids()`
- `place_bid()` RPC enforces same-tenant membership; cross-tenant bids
  rejected at the database layer
- Column-default routes authenticated frontend inserts to the caller's first
  membership so existing app code keeps working with zero code changes
- Prod-safe read-only verification script at `supabase/tests/multi_tenancy_verify.sql`

### Slice 6 — Onboarding wizard (commit `6b9a252`)

The closing piece. An approved operator clicks the link in their approval
email and walks through a 5-step wizard that ends with a live tenant + admin
account they can sign into.

- Migration `20260513000000_provision_tenant.sql`: `provision_tenant()` RPC
  SECURITY DEFINER. Atomically validates lead state + slug uniqueness, inserts
  the tenant row, inserts `tenant_members` with `role=admin`, marks the lead
  `status='onboarded'` and consumes the onboard_token. All three writes land
  or none do.
- Edge function `verify-onboard-token`: anon POST, 14-day TTL since
  `approved_at`, returns specific error codes (`invalid_token`,
  `token_expired`, `already_onboarded`, `lead_dropped`) so the UI can render a
  useful failure state
- Edge function `provision-tenant`: anon POST. Validates payload + token,
  pre-checks slug availability so we don't burn an auth user on a clash,
  creates auth user via `admin.createUser`, then calls the RPC. On RPC
  failure, best-effort `admin.deleteUser` cleanup. Returns `{tenant_id, slug,
  admin_email, redirect}`.
- `src/lib/onboard.ts`: typed `verifyOnboardToken()` + `provisionTenant()` +
  `slugify()` + `isValidSlug()`
- `OnboardWizard.tsx`: 5-step editorial form (Welcome / Identity / Mechanics /
  Admin / Review) with progress bar, per-step validation guard on Continue,
  prefilled values from the verified lead. Verify failure renders a focused
  error page; success state lays out tenant URL + admin email + sign-in CTA.

## What We Did Well

1. **Halfway assessment first, build second.** Spent 20 minutes mapping the
   actual ECTS evidence against the agreement before deciding what to build
   next. That made the SaPS pivot a deliberate choice — "the missing
   *Communicate* ECTS is the customer-visible half of the pivot, the missing
   *Research* ECTS is the case study" — instead of a vibe.

2. **Asked the architectural decisions upfront via `AskUserQuestion`.**
   Path-vs-subdomain routing and N:N-vs-1:1 membership were both hard to
   reverse. Locked them in 30 seconds with strong-recommendation defaults
   instead of committing to one path mid-implementation and regretting it.

3. **Parallel design specialists for the marketing surface.** Four agents
   (visual + layout + interaction + ux) returned in ~30 seconds with
   complementary specs. Implementation followed an opinionated brief rather
   than my own first instinct. The result feels intentionally editorial —
   credibility-shaped rather than SaaS-shaped — which matters more for B2B
   operator outreach than for the buyer marketplace.

4. **Atomic provisioning via SECURITY DEFINER RPC.** The "tenant exists"
   commitment crosses three rows (tenant + member + lead update). Doing those
   in JS would have left the door open to partial provisioning if the second
   write failed. The RPC makes it transactional at the DB layer with one
   round-trip from the edge function.

5. **Column-default + service-role audit instead of breaking changes.** Adding
   `NOT NULL tenant_id` to 10 transactional tables could have broken every
   running insert. The combination of `default_user_tenant()` for
   authenticated callers + an explicit `_shared/tenant.ts` helper for the 18
   service-role sites means the migration was safe to apply without
   coordinating a frontend deploy.

6. **Honest copy on the marketing surface.** No fabricated quotes, no fake
   logos, no "Trusted by 10,000+". The case study uses real numbers from the
   March 19 field visit, narrative voice, and observed behaviour. Trust
   signals (Paynow Integration #23997, dated audit, named contact) are all
   things that exist. This is the version we can actually defend in front of
   the panel.

## Great Ideas (for the remaining four weeks)

- **Wire Resend + actually invite three pilot operators.** The pipeline is
  built — running a real lead through it (cold visit → form → approve →
  onboard → first login) makes the SaPS pivot a demonstrable result rather
  than a built artifact. Three pilot tenants by June 8 is realistic.

- **Public status page at `/status` reading the existing security-agent +
  consistency-checker output.** Closes the SLA dimension of the Fingent
  framework. Half a day's work; reuses existing edge functions; visible signal
  of "professional service" to anyone evaluating the platform.

- **Tenant-aware navigation helper (`useTenantNavigate`)** so internal links
  preserve the `/t/<slug>` prefix automatically. Right now switching tenants
  is reset on the next `navigate('/payments')`. Deferred when only the demo
  tenant exists; lands the moment the second real tenant signs up.

- **Operator runbook + 10-minute training video as part of Slice 4 follow-up.**
  Fingent calls this "Training & Documentation". The wizard handles
  *Delivery*; the runbook handles *Support enablement*. PDF + screencap is
  faster than building a CSM portal and serves the same need for pilot scale.

- **Per-commit Lighthouse audit on `/operators` in CI.** Marketing pages are
  the first thing operators evaluate. A regression to PWA / accessibility /
  performance score should fail the build, not be discovered after deploy.

- **Audit log export RPC + an `/admin/audit?tenant=<slug>` view.** Several
  pricing-page promises ("immutable ledger you can show the regulator")
  imply this exists. The data exists in `settlement_ledger` and the
  notifications table — surfacing it as a downloadable CSV per tenant turns
  a copy promise into a delivered feature.
